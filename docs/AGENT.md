# توثيق HR Agent — الوكيل الذكي والتقنيات المستخدمة

> هذا المستند يوثّق البنية التقنية لمشروع **HR Agent**: كيف يعمل الوكيل الذكي (AI Agent) داخلياً،
> وما هي التقنيات والمكتبات المستخدمة في بناء الباك-إند والواجهة الأمامية.

---

## 1. نظرة عامة على المشروع

HR Agent هو نظام يساعد فرق الموارد البشرية على:

1. **تحليل السير الذاتية** للمتقدمين وتسجيلها مقابل معايير تقييم موزونة لكل مجال وظيفي.
2. **توليد أسئلة مقابلة** (مفتوحة أو اختيار من متعدد) مبنية على محتوى السيرة الذاتية الفعلي.
3. **إجراء مقابلة صوتية آلية** مع المرشح عبر المتصفح (تحويل كلام إلى نص وقراءة الأسئلة صوتياً)، ثم تقييمها بالذكاء الاصطناعي.
4. **تصحيح اختبار ورقي/كتابي** تكتب فيه HR إجابات المرشح المكتوبة يدوياً ليصحّحها النظام.
5. **توليد توصية نهائية** تجمع كل مصادر التقييم المتاحة (سيرة ذاتية + مقابلة صوتية + اختبار كتابي).

الفلسفة الأساسية (مذكورة في `README.md`): **الإنسان في الحلقة (Human-in-the-loop)** — الوكيل يقترح
ويُرتّب فقط، والقرار النهائي دائماً بيد موظف HR. كل واجهات النظام والمخرجات باللغة العربية (RTL).

لا توجد قاعدة بيانات في المشروع؛ **كل التخزين عبارة عن ملفات JSON على القرص**، ملف لكل سجل
(طلب توظيف / وظيفة / موظف).

## 2. المعمارية العامة

```
                          ┌─────────────────────────┐
                          │   المتصفح (Frontend)     │
                          │  HTML + CSS + JS خالص    │
                          │  بدون أي framework       │
                          └───────────┬─────────────┘
                                      │ fetch() / REST
                                      ▼
                          ┌─────────────────────────┐
                          │   FastAPI  (main.py)     │
                          │   /api/v1/*  routes.py   │
                          └───────────┬─────────────┘
                                      │
              ┌───────────────────────┼───────────────────────┐
              ▼                       ▼                       ▼
     ┌─────────────────┐   ┌──────────────────────┐  ┌──────────────────┐
     │  services/*.py   │   │  criteria/*.json      │  │  data/*.json       │
     │  منطق العمل       │   │  معايير التقييم        │  │  التخزين (Storage) │
     │  (تحليل، أسئلة،   │   │  لكل مجال وظيفي        │  │  applications/     │
     │  مقابلة، اختبار،  │   └──────────────────────┘  │  jobs/             │
     │  توصية)           │                              │  employees/         │
     └────────┬─────────┘                              └──────────────────┘
              │
              ▼
     ┌─────────────────────┐
     │   llm_client.py      │   ← نقطة الوصل الوحيدة إلى الذكاء الاصطناعي
     │  generate_text()      │
     └──────────┬───────────┘
                │
        ┌───────┴────────┐
        ▼                ▼
 ┌─────────────┐   ┌──────────────┐
 │  Anthropic    │   │  Google Gemini │
 │  Claude API   │   │      API       │
 └─────────────┘   └──────────────┘
```

**نقطة التصميم الأهم:** كل خدمات الأعمال (تحليل السيرة، توليد الأسئلة، تقييم المقابلة الصوتية،
تصحيح الاختبار، التوصية النهائية) لا تتحدث مباشرة مع Anthropic أو Gemini — بل تمر جميعها عبر
دالة واحدة `generate_text()` في `llm_client.py`. هذا يجعل تبديل مزوّد الذكاء الاصطناعي (Claude ↔
Gemini) مجرد تغيير في متغيرات البيئة، بدون أي تعديل في الكود.

## 3. التقنيات المستخدمة (Tech Stack)

### 3.1 الباك-إند (Backend)

| التقنية | الإصدار/الاستخدام | الغرض |
|---|---|---|
| **Python** | 3.13 | لغة التطوير |
| **FastAPI** | `>=0.110` | إطار عمل الـ API (REST) |
| **Uvicorn** | `>=0.29` (مع extras `[standard]`) | سيرفر ASGI لتشغيل FastAPI |
| **Pydantic** | `>=2.6` | التحقق من صحة البيانات (Schemas) للطلبات والاستجابات |
| **python-dotenv** | `>=1.0` | تحميل متغيرات البيئة من ملف `.env` |
| **python-multipart** | `>=0.0.9` | استقبال رفع الملفات (multipart/form-data) |
| **pytest** | `>=8.0` | تشغيل الاختبارات |

### 3.2 الذكاء الاصطناعي (AI Providers)

| التقنية | الاستخدام |
|---|---|
| **anthropic SDK** (`>=0.39`) | استدعاء نماذج **Claude** (المزوّد الافتراضي) — النموذج الافتراضي `claude-sonnet-4-5` |
| **google-genai SDK** (`>=0.9`) | استدعاء نماذج **Gemini** (مزوّد بديل) — النموذج الافتراضي `gemini-2.5-flash` |

يتم اختيار المزوّد عبر متغير البيئة `HR_AGENT_PROVIDER` (`anthropic` أو `gemini`)، بدون أي تغيير
في منطق التطبيق.

### 3.3 استخراج ومعالجة المستندات

| التقنية | الغرض |
|---|---|
| **pypdf** (`>=4.0`) | استخراج النص من ملفات PDF (السيرة الذاتية) |
| **python-docx** (`>=1.1`) | استخراج النص من ملفات Word (.docx) |
| **beautifulsoup4** (`>=4.12`) | استخراج النص من صفحات HTML عند تقديم رابط بدل ملف |
| **httpx** (`>=0.27`) | تنفيذ طلبات HTTP لجلب السيرة الذاتية من رابط خارجي |
| **openpyxl** (`>=3.1`) | قراءة وكتابة ملفات Excel (استيراد/تصدير بيانات الموظفين) |

### 3.4 الواجهة الأمامية (Frontend)

- **HTML + CSS + JavaScript خالص (Vanilla)** — لا يوجد أي إطار عمل (لا React ولا Vue ولا Alpine)
  ولا أي مكتبة محمّلة من CDN، ولا خطوة بناء (build step).
- تصميم **RTL** بالكامل (`dir="rtl"`) مع دعم الوضع الفاتح/الداكن عبر `prefers-color-scheme`.
- استدعاءات الـ API تتم عبر `fetch()` مباشرة.
- **Web Speech API** المدمجة في المتصفح (Chrome/Edge) لصفحة المقابلة الصوتية:
  - `SpeechRecognition` / `webkitSpeechRecognition` (`ar-SA`) لتحويل كلام المرشح إلى نص.
  - `speechSynthesis` / `SpeechSynthesisUtterance` (`ar-SA`) لقراءة الأسئلة صوتياً.
  - لا يوجد أي خدمة خارجية للتحويل الصوتي — كل شيء يعتمد على محرك المتصفح نفسه.

### 3.5 التخزين (Storage)

**لا توجد قاعدة بيانات.** التخزين بالكامل ملفات JSON مسطّحة على القرص:

- `backend/app/data/applications/*.json` — طلبات التوظيف.
- `backend/app/data/jobs/*.json` — المسميات الوظيفية.
- `backend/app/data/employees/*.json` — بيانات الموظفين.

كل سجل يُخزَّن في ملف مستقل باسم مبني على `uuid.uuid4().hex[:12]`، وتُدار عبر وحدات "store" بسيطة
(`application_store.py`, `job_store.py`, `employee_store.py`) توفر عمليات إنشاء/قراءة/تحديث/حذف
(CRUD) عبر قراءة/كتابة الملفات مباشرة.

## 4. الوكيل الذكي (AI Agent) — كيف يعمل

### 4.1 طبقة التجريد: `llm_client.py`

نقطة الدخول الوحيدة هي:

```python
generate_text(prompt: str, max_tokens: int = 2000) -> str
```

- تتحقق أولاً أن المزوّد مُهيّأ (`settings.is_configured`)، وإلا ترفع `LLMError`.
- توجّه الاستدعاء حسب `settings.PROVIDER`:
  - **Anthropic:** `Anthropic(api_key=...).messages.create(model=settings.MODEL, ...)`
  - **Gemini:** `genai.Client(api_key=...).models.generate_content(model=settings.GEMINI_MODEL, ...)`
- تُرجع نصاً خاماً (يُفترض أن يكون JSON)، ثم تُمرَّر النتيجة إلى `extract_json()`
  (في `json_utils.py`) التي تُزيل أي "code fences" من نوع ` ```json ... ``` ` وتُحوّلها إلى `dict`
  عبر `json.loads`.

### 4.2 الـ Prompts (`core/prompts.py`)

جميع الـ prompts مكتوبة بالعربية، وتطلب من النموذج الرد **بصيغة JSON فقط** (بدون أي markdown)،
وتحتوي كل واحدة منها على تعليمات صريحة لمنع التحيّز ("تجنّب أي تحيّز متعلق بالجنس أو الجنسية أو
العمر أو العرق — قيّم الكفاءة فقط") وتذكير بأن الناتج توصية لموظف HR وليس قراراً نهائياً.

يوجد 6 منشئات (builders) للـ prompts، كل منها يخدم ميزة محددة:

| الدالة | الاستخدام | ناتج JSON المتوقع |
|---|---|---|
| `build_cv_analysis_prompt` | تحليل السيرة الذاتية مقابل معايير المجال | `{scores:[{key, score, justification_ar}], strengths_ar, gaps_ar, recommendation_ar}` |
| `build_interview_questions_prompt` | توليد 6-8 أسئلة مقابلة مفتوحة (تقنية + سلوكية STAR) | `{questions: ["..."]}` |
| `build_mcq_questions_prompt` | توليد 5-8 أسئلة اختيار من متعدد (4 خيارات لكل سؤال) | `{questions:[{text, options[4], correct_answer}]}` |
| `build_voice_evaluation_prompt` | تقييم نص المقابلة الصوتية على 4 أبعاد | `{scores:[{key, score, justification_ar}×4], overall_summary_ar}` |
| `build_written_test_prompt` | تصحيح إجابات الاختبار الكتابي المفتوحة | `{results:[{index, score, justification_ar}]}` |
| `build_final_recommendation_prompt` | توليد التوصية النهائية من كل مصادر التقييم المتاحة | `{recommend: bool, reason_ar, strengths_ar[], weaknesses_ar[]}` |

### 4.3 سير عمل كل ميزة من ميزات الوكيل

#### أ) تحليل السيرة الذاتية — `cv_analyzer.py`

1. تحميل معايير المجال الوظيفي عبر `load_criteria(domain)`.
2. بناء الـ prompt وإرساله للنموذج عبر `generate_text`.
3. تحليل رد النموذج JSON عبر `extract_json`.
4. **حساب الدرجة الإجمالية الموزونة يدوياً في بايثون** (وليس بالاعتماد على حساب النموذج):
   لكل معيار في ملف المعايير، `weighted_total += raw_score * weight / 100` — هذا يمنع النموذج من
   التلاعب بالنتيجة النهائية عبر اختراع أو حذف معايير.
5. إرجاع نتيجة تحتوي: الدرجة الإجمالية، التوصية، تفاصيل كل معيار، نقاط القوة والفجوات، وإخلاء
   مسؤولية ثابت (disclaimer).

#### ب) توليد أسئلة المقابلة — `interview_questions.py`

- يختار بين نوعين: أسئلة مفتوحة أو اختيار من متعدد (`question_type`).
- يتحقق من صحة كل سؤال قبل قبوله (نص غير فارغ، خيارات غير فارغة لأسئلة الاختيار)، ويرفض السؤال
  التالف بدل فشل العملية بالكامل.

#### ج) المقابلة الصوتية — `voice_interview.py` + صفحة `interview.html`

- المرشح يفتح صفحة المقابلة، يقرأ النظام السؤال صوتياً عبر `speechSynthesis`، ثم يسجّل إجابته عبر
  `SpeechRecognition` (تحويل كلام إلى نص مباشرة في المتصفح، بدون أي خدمة خارجية).
- بعد كل الأسئلة، يُرسل النص المُجمّع (transcript) إلى `evaluate_voice_interview` التي:
  - تبني prompt من النص الكامل وترسله للنموذج.
  - **تفرض هيكلاً ثابتاً من 4 أبعاد تقييم** بغض النظر عمّا يُرجعه النموذج:
    الثقة (confidence)، الثقافة العامة والمهنية (culture)، الفهم (understanding)،
    المنطق والتحليل (reasoning) — أي بُعد مفقود من رد النموذج يُعطى درجة 0 تلقائياً.

#### د) الاختبار الكتابي/الورقي — `written_test.py`

- تصحيح **هجين**:
  - أسئلة الاختيار من متعدد تُصحَّح **محلياً بدون أي استدعاء للنموذج** (مطابقة نصية مباشرة =
    100 أو 0).
  - الأسئلة المفتوحة تُجمَّع كلها في استدعاء واحد للنموذج (وليس استدعاء لكل سؤال) لتوفير التكلفة
    وتسريع الاستجابة.
- الدرجة الإجمالية = متوسط كل الإجابات المصحّحة.

#### هـ) التوصية النهائية — `recommendation.py`

- تتطلب توفّر مصدر تقييم واحد على الأقل (سيرة ذاتية / مقابلة صوتية / اختبار كتابي).
- تبني ملخصاً من كل المصادر المتاحة، وتطلب من النموذج توصية نهائية شاملة نقاط القوة والضعف.
- القرار الفعلي (قبول/رفض) يبقى دائماً فعلاً بشرياً منفصلاً (`POST /decision`)، والتوصية مجرد مُدخَل
  مساعِد لموظف HR.

## 5. معايير التقييم (`criteria/*.json`)

كل مجال وظيفي له ملف JSON مستقل (حالياً: `accounting.json` و `engineering.json`) بالشكل:

```json
{
  "domain": "accounting",
  "domain_ar": "المحاسبة",
  "weights_sum_to": 100,
  "criteria": [
    {"key": "education", "label_ar": "المؤهل العلمي", "weight": 15, "signals": [...]},
    {"key": "certifications", "label_ar": "الشهادات المهنية", "weight": 20, "signals": ["SOCPA","CPA","CMA", "..."]}
  ]
}
```

- يُتحقق أن مجموع الأوزان يساوي `weights_sum_to` (عادة 100)، وأن كل معيار يحتوي `key` و `label_ar`
  و `weight` (`criteria_loader.py`).
- هذا الملف هو **نقطة التوسّع الأساسية**: يمكن لخبير HR تعديله مباشرة (JSON بسيط) بدون الحاجة إلى
  مبرمج، أو لاحقاً اشتقاق الأوزان من بيانات توظيف فعلية (dataset) بدل التعيين اليدوي.

## 6. نقاط النهاية (API Endpoints)

كل المسارات تحت البادئة `/api/v1`. المسارات المعلّمة **[Admin]** تتطلب تسجيل دخول HR (كوكي جلسة
`hr_session`)؛ الباقي عام (لصفحات المرشح الخارجية).

| المجموعة | أمثلة على المسارات |
|---|---|
| **المصادقة** | `POST /auth/login`, `POST /auth/logout`, `GET /auth/status`, `POST /auth/change-password` [Admin] |
| **استخراج النص** | `POST /extract/file` [Admin], `POST /extract/url` [Admin] |
| **المجالات والمعايير** | `GET /domains`, `GET /criteria/{domain}` |
| **الوظائف** | `GET /jobs`, `POST /jobs` [Admin], `PUT /jobs/{id}` [Admin], `DELETE /jobs/{id}` [Admin] |
| **الموظفون** | `GET/POST/PUT/DELETE /employees` [Admin], `GET /employees/import/template` [Admin], `POST /employees/import` [Admin] |
| **تحليل السيرة/الطلبات** | `POST /analyze` [Admin], `POST /apply` (عام), `GET /applications` [Admin], `GET /applications/status` (عام), `GET /applications/{id}` [Admin], `POST /applications/{id}/approve|reject` [Admin] |
| **أسئلة المقابلة** | `POST /applications/{id}/questions/generate` [Admin], `POST/PUT/DELETE /applications/{id}/questions/{qid}` [Admin] |
| **المقابلة الصوتية** | `GET /applications/{id}/interview/questions` (عام، بشرط الأهلية), `POST .../interview/turn`, `POST .../interview/evaluate` |
| **الاختبار الكتابي** | `POST /applications/{id}/written-test/submit` [Admin] |
| **التوصية والقرار** | `POST /applications/{id}/recommendation/generate` [Admin], `POST /applications/{id}/decision` [Admin] |

## 7. الأمان

- **جلسات بسيطة داخل الذاكرة (in-memory sessions)** بدل JWT أو قاعدة بيانات: توكن عشوائي
  (`secrets.token_urlsafe(32)`) صالح لمدة 12 ساعة، يُفقد عند إعادة تشغيل السيرفر. مناسب لحساب HR
  إداري واحد فقط — أول شيء يجب استبداله عند الحاجة لمستخدمين متعددين.
- كلمة المرور تُقارَن بطريقة **مقاومة لهجمات التوقيت** (`secrets.compare_digest`)، وتغييرها يُحفظ
  مباشرة في ملف `.env`.
- **حماية من هجمات SSRF** عند استخراج السيرة الذاتية من رابط (`extract_text_from_url`): يُرفض أي
  IP خاص/محلي/محجوز، تُرفض إعادة التوجيه (redirects)، ويُشترط بروتوكول http/https فقط.
- الملفات المرفوعة محدودة بحجم 5MB، والنص المستخرج يُقصّ عند 20,000 حرف.

## 8. التشغيل والإعداد

```bash
cd backend
pip install -r requirements.txt
cp .env.example .env     # ثم عبّئ المفاتيح
uvicorn app.main:app --reload
```

أهم متغيرات البيئة (`backend/.env`):

| المتغير | الافتراضي | الوصف |
|---|---|---|
| `HR_AGENT_PROVIDER` | `anthropic` | مزوّد الذكاء الاصطناعي (`anthropic` أو `gemini`) |
| `ANTHROPIC_API_KEY` | — | مفتاح Anthropic |
| `HR_AGENT_MODEL` | `claude-sonnet-4-5` | نموذج Claude المستخدم |
| `GEMINI_API_KEY` | — | مفتاح Google Gemini |
| `HR_AGENT_GEMINI_MODEL` | `gemini-2.5-flash` | نموذج Gemini المستخدم |
| `HR_AGENT_DOMAIN` | `accounting` | المجال الوظيفي الافتراضي |
| `HR_ADMIN_USERNAME` / `HR_ADMIN_PASSWORD` | `admin` / `changeme123` | بيانات دخول HR |

بعد التشغيل:
- `/app/` — لوحة تحكم HR (تتطلب دخول عبر `/login`).
- `/apply` — صفحة تقديم عامة للمرشحين.
- `/interview` — صفحة المقابلة الصوتية للمرشح.
- `/health` — حالة النظام والمزوّد النشط.

## 10. نقاط التوسّع المستقبلي

- **معايير التقييم (`criteria/`)**: قابلة للتعديل من خبير HR مباشرة، أو لاحقاً اشتقاقها من dataset
  توظيف حقيقي.
- **مزوّد الذكاء الاصطناعي**: التبديل بين Claude و Gemini (أو إضافة مزوّد جديد) يتطلب فقط تعديل
  `llm_client.py` في مكان واحد، دون المساس بخدمات الأعمال الأخرى.
- **المصادقة**: النظام الحالي بحساب إداري واحد فقط داخل الذاكرة — التوسّع لمستخدمين متعددين يتطلب
  استبدال `services/auth.py` بنظام مصادقة حقيقي (قاعدة بيانات + كلمات مرور مشفّرة لكل مستخدم).
